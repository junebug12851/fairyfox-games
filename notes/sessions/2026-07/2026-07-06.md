## FIX: kill the white flash when opening a game (v0.18.1)

Owner reported a white flash **every time** you open a game from the hub home, but never when
navigating between hub pages. Diagnosis: cross-document navigation shows a brief blank frame
(Chrome's paint-holding gap); the hub/landing pages ship `<meta name="theme-color">` (dark) so they
never expose white, but the 11 game shells carried no early colour signal — their dark background
lived only inside the inline `<style>` as `background:var(--bg)` — so the gap fell back to the
browser's default white. Not a game-logic issue.

Fix (pure static HTML, all 11 shells): added a literal `background:#<bg>` on the `<html>` element
**and** a matching `<meta name="theme-color" content="#<bg>">`, using each game's own `--bg`. The
first painted frame is now dark regardless of when the `<style>` parses. Render-identical (same
colour, declared earlier) — zero visual delta, cores/tests untouched.

Also clarified for the owner (they asked): the site is — correctly — plain static HTML/CSS/vanilla
JS, with **no Jekyll** (`.nojekyll` is intentional so Pages serves files verbatim) and **no
React**; neither is involved, since the flash is a browser paint gap, not a build/render concern.

Tests: collection **361/361** green (HTML-only change). Preview: served over `http://localhost`,
Chrome-checked Arc (`#0a0a12`) and Orbit Slingshot (`#06060d`) — both render clean and dark, no
regression. Released `dev → main` by default on green (PATCH v0.18.1).

## PLANT: new game **Sluice** — colour-sorting (a new verb: sort/route) (v0.18.0)

Automated PLANT run (`fairyfox-games-new`). Started clean on `dev`, up to date; no open
issues/PRs to triage.

**Design.** Needed a mechanically-distinct verb. Existing verbs: steer, time-a-catch, thrust,
flip-match, aim-bounce, precision-stack, keep-aloft/rhythm, balance, mirror-coordination,
judge-power. Picked **sort/route** — categorise a falling item into the matching bin — which none
of the ten cover. Made it a genuine *read* (not muscle memory) by having the channels rearrange,
so the correct slot moves even though the spark's colour doesn't. Kept the on-ramp trivial
(3 fixed channels, slow fall, calm Steady formation) so it's ~3s to grasp, with depth unlocking.

**Core-fun.** The hook is a *snap combo*: route early (within `SNAP_FRAC` of the fall) → the
multiplier climbs; route slow-but-correct → scores without growing it. So under the scrambling
patterns you choose per spark between a fast trust-read (grow the combo, risk a misread) and a
sure slow-read (bank the score, stall the combo). Lives = 3; a wrong channel or a landed spark is
a miss and breaks the combo.

**Build (full bar, copied in shape from Polarity/Ink Bloom).** Pure `sluice.core.js` — plain data
+ pure fns, injectable RNG, no DOM. Varied structure: 7 stage-weighted `FORMATIONS` (Steady, Run,
Alternate calm; Shuffle, Cascade, Rush, The Churn notable) gated by `minStage`, `pickFormation`/
`loadFormation`, `spawnDrop` pulls per-formation specs `{color, shuffle, fast}`. Stages
Trickle→Stream→Rapids→Cataract→Maelstrom speed the fall *and* widen the channels (3→4 via
`binsAt`). `permuteBins` guarantees a different arrangement. Meta `sluice.meta` (lifetime
sorts/snaps, bestStage/bestMult, 10 skill-safe badges, run-report), legacy `sluice.best`
preserved. Shell = external module + classic-script boot fallback, guarded loop, error/rejection
logging, reduced-motion-safe cue, number-key + tap input, lives pips, channel flashes, miss splash.
Icon drawn (System.Drawing).
