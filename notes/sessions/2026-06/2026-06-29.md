## Ink Bloom boot fix — v0.1.1

A real in-browser play-check (local server + automated steering + screenshots) caught
that the game didn't actually boot on a normal page load, even though its logic was
fully unit-tested and ran fine when injected post-load. Root cause: the render shell
was an inline `<script type="module">` with a top-level `await import()`, which is
fragile to boot as the document's own module. Fix: extracted the shell to an external
module (`ink-bloom.shell.js`) loaded via `<script type="module" src>` — the
conventional structure — which boots reliably. Confirmed live: boots on clean load,
steers, plays. Added error handling along the way (boot-failure fallback message,
guarded render loop, global error/rejection logging) so a silently-dead screen can't
recur. Released v0.1.1.

node. Decided the **collection** is the project (not one repo per game), so this is a
monorepo of self-contained game folders.

  JSDoc'd, no DOM) from the rendering shell, and wrote a 20-test suite. Found + fixed
  the original prototype's frame-one death bug (the starting trail was seeded with the
  newest point at index 0, putting an old collidable point on the head); added a
  regression test.
- Scaffolded the repo: landing page, README (contributions welcome), CONTRIBUTING,
  MIT, issue/PR templates, CI (`node --test`), Pages deploy workflow, notes system,
  mesh-aware `CLAUDE.md`.

  registries; added a node page.
